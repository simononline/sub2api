# My Sub2API Docker 发布构建方案

本文档梳理当前项目改为 `my-sub2api` 命名后的本地镜像发布流程。

## 一、推荐发布形态

当前项目原始官方构建方式是根目录 `Dockerfile` 的单镜像模式：先构建 Vue 前端，再把前端产物嵌入 Go 后端，最终发布一个 `sub2api` 应用镜像。

本项目本地发布改为四个镜像：

| 类型       | 镜像名                          | 更新频率     | 处理策略                     |
| ---------- | ------------------------------- | ------------ | ---------------------------- |
| PostgreSQL | `my-sub2api-postgres:18-alpine` | 很低         | 只在本地不存在时创建         |
| Redis      | `my-sub2api-redis:8-alpine`     | 很低         | 只在本地不存在时创建         |
| Backend    | `my-sub2api-backend:<TAG>`      | 每次业务发布 | 如果已存在则删除，再重新构建 |
| Frontend   | `my-sub2api-frontend:<TAG>`     | 每次业务发布 | 如果已存在则删除，再重新构建 |

这样做的原因：

- PostgreSQL 和 Redis 是基础服务镜像，不应该每次业务发版都重建。
- 后端和前端是业务代码镜像，每次发布都应明确重建，避免旧镜像误用。
- 后端不再使用 `-tags embed` 嵌入前端；前端由独立 Nginx 镜像提供静态资源。
- 前端 Nginx 代理 `/api`、`/v1`、`/setup`、`/health` 到后端，浏览器仍按同源访问，不需要额外跨域配置。

## 二、新增构建文件

本流程使用以下文件：

- `deploy/release-images.sh`：本地发布镜像脚本
- `deploy/Dockerfile.backend`：只构建 Go 后端镜像
- `deploy/Dockerfile.frontend`：构建 Vue 前端并用 Nginx 提供服务
- `deploy/nginx.my-sub2api.conf`：前端 Nginx 静态资源和 API 代理配置
- `deploy/docker-compose.my-sub2api.yml`：使用拆分镜像在本地运行完整服务
- `deploy/docker-entrypoint.sh`：运行入口，已支持通过 `APP_USER` 指定容器内运行用户

原有根目录 `Dockerfile` 仍可用于官方单镜像构建，但本方案优先使用新增的拆分镜像流程。

## 三、发布脚本执行顺序

执行脚本：

```bash
./deploy/release-images.sh
```

脚本会按顺序执行：

1. 检查 `my-sub2api-postgres:18-alpine`
   - 如果已存在：跳过
   - 如果不存在：拉取 `postgres:18-alpine` 并打本地 tag

2. 检查 `my-sub2api-redis:8-alpine`
   - 如果已存在：跳过
   - 如果不存在：拉取 `redis:8-alpine` 并打本地 tag

3. 检查 `my-sub2api-backend:<TAG>`
   - 如果已存在：先删除
   - 然后基于 `deploy/Dockerfile.backend` 重新构建

4. 检查 `my-sub2api-frontend:<TAG>`
   - 如果已存在：先删除
   - 然后基于 `deploy/Dockerfile.frontend` 重新构建

## 四、默认镜像名

默认执行：

```bash
./deploy/release-images.sh
```

等价于：

```text
PROJECT_NAME=my-sub2api
TAG=local
SQL_IMAGE=my-sub2api-postgres:18-alpine
REDIS_IMAGE=my-sub2api-redis:8-alpine
BACKEND_IMAGE=my-sub2api-backend:local
FRONTEND_IMAGE=my-sub2api-frontend:local
```

发布指定 tag：

```bash
TAG=0.1.119-local ./deploy/release-images.sh
```

自定义项目名前缀：

```bash
PROJECT_NAME=my-sub2api TAG=0.1.119-local ./deploy/release-images.sh
```

## 五、版本和构建参数

后端镜像支持这些构建参数，脚本会自动传入：

- `VERSION`：业务版本；未指定时后端构建会读取 `backend/cmd/server/VERSION`
- `COMMIT`：默认读取当前 Git 短提交号，读取失败时使用 `docker`
- `DATE`：默认使用当前 UTC 时间
- `GOPROXY`：默认 `https://goproxy.cn,direct`
- `GOSUMDB`：默认 `sum.golang.google.cn`

示例：

```bash
VERSION=0.1.119-my \
TAG=0.1.119-my \
GOPROXY=https://proxy.golang.org,direct \
GOSUMDB=sum.golang.org \
./deploy/release-images.sh
```

## 六、前后端拆分后的运行关系

容器运行时建议服务名保持：

- 后端服务名：`backend`
- 前端服务名：`frontend`
- PostgreSQL 服务名：`postgres`
- Redis 服务名：`redis`

原因是 `deploy/nginx.my-sub2api.conf` 中前端会代理到：

```text
http://backend:8080
```

后端环境变量中应配置：

```text
DATABASE_HOST=postgres
DATABASE_PORT=5432
REDIS_HOST=redis
REDIS_PORT=6379
AUTO_SETUP=true
```

前端只需要暴露 Nginx 的 `80` 端口，默认映射到宿主机 `8080`：

```text
8080:80
```

浏览器访问：

```text
http://localhost:8080
```

使用专用 Compose 文件启动：

```bash
cd deploy
cp .env.example .env
mkdir -p data postgres_data redis_data
docker compose -f docker-compose.my-sub2api.yml up -d
```

查看日志：

```bash
docker compose -f docker-compose.my-sub2api.yml logs -f backend
```

## 七、本地验证命令

构建所有镜像：

```bash
./deploy/release-images.sh
```

查看镜像：

```bash
docker images | grep my-sub2api
```

只查看某个镜像是否存在：

```bash
docker image inspect my-sub2api-backend:local
docker image inspect my-sub2api-frontend:local
```

清理业务镜像后重新发布：

```bash
docker rmi my-sub2api-backend:local my-sub2api-frontend:local
./deploy/release-images.sh
```

注意：脚本本身每次都会重建 backend/frontend，所以通常不需要手动清理业务镜像。

## 八、注意事项

1. SQL 和 Redis 镜像只是基础镜像本地别名，不包含业务数据。
2. 业务数据仍应通过 Docker volume 或宿主机目录挂载保存。
3. 后端二进制内仍然嵌入 SQL migrations，应用启动后会按 `schema_migrations` 记录执行数据库迁移。
4. 拆分前后端后，后端镜像不再服务 SPA 页面；浏览器入口应访问前端 Nginx。
5. 如果修改了 PostgreSQL 或 Redis 主版本，再更新 `deploy/release-images.sh` 中的默认基础镜像版本和本地镜像 tag。




# 手动执行的步骤记录(可以使用下方一句话命令)
## 第一步命令
必须限制性这个：

```bash
命令：
./deploy/release-images.sh

返回结果
View build details: docker-desktop://dashboard/build/orbstack/orbstack/74v2128t8uvbmgvssidvr3xs6
[release-images] 镜像创建完成:
[release-images]   SQL      my-sub2api-postgres:18-alpine
[release-images]   Redis    my-sub2api-redis:8-alpine
[release-images]   Backend  my-sub2api-backend:local
[release-images]   Frontend my-sub2api-frontend:local
```

#### 直接启动
```bash
docker compose --env-file ./deploy/.env -f ./deploy/docker-compose.my-sub2api.yml up -d
```

#### 查看容器状态
```bash
docker compose --env-file ./deploy/.env -f ./deploy/docker-compose.my-sub2api.yml ps
```
#### 查看后端日志
```bash
docker compose --env-file ./deploy/.env -f ./deploy/docker-compose.my-sub2api.yml logs -f backend
```
#### 停止服务
```bash
docker compose --env-file ./deploy/.env -f ./deploy/docker-compose.my-sub2api.yml down
```
#### 如果你想重新发布新代码并重启：  （注意在根目录）
```bash
cd ..
./deploy/release-images.sh
cd deploy
docker compose --env-file ./deploy/.env -f ./deploy/docker-compose.my-sub2api.yml up -d
```

#### 如果想删除docker的容器
```bash
# 替换掉关键词：my-sub2api:latest  复制并直接执行

# 找到容器
docker ps -a --filter ancestor=my-sub2api:latest

# 停止容器
docker ps -q --filter ancestor=my-sub2api:latest | xargs -r docker stop

# 删除容器
docker ps -aq --filter ancestor=my-sub2api:latest | xargs -r docker rm -f

# 删除镜像
docker rmi -f my-sub2api:latest

```
