# 支付方案调研

更新时间：`2026-04-29`

## 结论摘要

- 业务侧建议：在 `中国大陆主体 + H5 + 海内外用户各半 + 先快速上线` 的前提下，优先评估 `PingPong`，备选 `Oceanpayment`。
- 项目侧建议：不要直接全面替换现有支付体系，采用 `新增并切流` 更稳妥。
- 复杂度判断：
  - `新增中国钱包通道并复用现有 alipay/wxpay 入口`：`中等`
  - `替换 Stripe 的国际卡/订阅购付费链路`：`中高`
  - `全面替换现有支付体系`：`高`

如果目标是“尽快同时满足中国境内支付宝/微信/银行卡，以及海外银行卡”，当前项目最现实的路线是：

1. 新增一个聚合支付 provider，优先承接中国钱包。
2. 用户侧继续展示现有 `alipay` / `wxpay`，不改用户心智。
3. 国际卡先保留现有 Stripe 链路，后续再决定是否替换。

## 渠道推荐

### 1. 主推荐：PingPong

适合原因：

- 更贴近 `中国大陆主体` 的跨境收单场景。
- 官方资料覆盖跨境收单与多支付方式，适合同时承接中国钱包和国际卡。
- 对你当前项目来说，更适合先作为“新增 provider + 渐进切流”的第一站。

推荐接入方式：

- Phase 1 先接入中国钱包能力，继续对前端暴露现有 `alipay` / `wxpay`。
- 如果 PingPong 后续确认国际卡 Hosted Checkout、退款、查单、Webhook、争议处理能力都满足，再决定是否逐步替换部分 Stripe 能力。

### 2. 备选：Oceanpayment

适合原因：

- 也是偏跨境收单和本地支付聚合的路线。
- 如果 `API 额度充值 / 会员套餐 / 虚拟服务` 这类品类在首选渠道审核不顺，Oceanpayment 是合理备选。
- 对大陆主体来说，落地方式与“新增并切流”策略兼容。

### 3. 为什么不建议首版追求单通道统一

- 当前最贵的不是“多一个渠道”，而是“把现有支付语义全部重写”。
- 首版如果强行追求一个渠道同时覆盖中国钱包、国际卡、未来自动续费、退款、Webhook、后台配置，会把技术和业务审核风险叠加到同一轮上线里。

### 4. 为什么暂不建议把 Airwallex / Stripe 当作本文 v1 主选

- `Stripe` 更适合海外主体与国际卡优先场景；对你当前的大陆主体和中国钱包优先目标，不是最短路径。
- `Airwallex` 更适合香港、新加坡或海外主体的统一收款设计；本文场景下不作为 v1 主选，但未来若主体结构变化，仍值得重新评估。

## 当前项目支付架构判断

### 可插拔基础已经存在

当前仓库并不是“把支付写死在一个 SDK 里”，而是已经有清晰的 provider 抽象，说明“新增一个 provider，再逐步切流”是可行的。

- Provider 抽象和能力边界已经独立定义在 [../backend/internal/payment/types.go](../backend/internal/payment/types.go)
  - 包含 `CreatePayment / QueryOrder / VerifyNotification / Refund` 等统一接口。
- Provider 工厂是显式 `provider_key` 分发，在 [../backend/internal/payment/provider/factory.go](../backend/internal/payment/provider/factory.go)
  - 当前支持 `easypay / alipay / wxpay / stripe`。
- Provider 实例配置通过数据库管理，加载逻辑在 [../backend/internal/service/payment_service.go](../backend/internal/service/payment_service.go)
  - 说明新增渠道不是改单一环境变量，而是纳入现有实例化体系。
- 订单表已保存 `provider_instance_id / provider_key / provider_snapshot`，见 [../backend/ent/schema/payment_order.go](../backend/ent/schema/payment_order.go)
  - 这让历史订单、退款、Webhook 对应关系更容易兼容。

这一层的结论是：`基础架构支持新增 provider`。

### 替换成本主要不在订单表，而在“支付语义硬编码”

当前替换成本更大的部分，是前后端都对部分支付方式和 provider key 做了显式分支。

- 前端可见支付方式目前主要围绕 `alipay / wxpay / stripe` 组织，见 [../frontend/src/components/payment/paymentFlow.ts](../frontend/src/components/payment/paymentFlow.ts)
  - `normalizeVisibleMethod` 和支付跳转决策都围绕这几个值展开。
- Provider 配置、Webhook 路径、支持类型映射写死在 [../frontend/src/components/payment/providerConfig.ts](../frontend/src/components/payment/providerConfig.ts)
  - `PROVIDER_SUPPORTED_TYPES`
  - `WEBHOOK_PATHS`
  - `PROVIDER_CONFIG_FIELDS`
- 前端类型也把支付方式写成固定 union，见 [../frontend/src/types/payment.ts](../frontend/src/types/payment.ts)
  - 新增真正的 first-class 支付方式，需要同步更新类型、配置和页面行为。
- Webhook 路由是显式注册的，见 [../backend/internal/server/routes/payment.go](../backend/internal/server/routes/payment.go)
  - 当前只有 `easypay / alipay / wxpay / stripe`。
- Webhook 处理器同样按 provider key 分发，见 [../backend/internal/handler/payment_webhook_handler.go](../backend/internal/handler/payment_webhook_handler.go)
  - 新 provider 若需要独立回调路径，需要后端路由和处理器同步扩展。
- 后端“可见支付方式来源”目前主要解析 `alipay / wxpay`，见 [../backend/internal/service/payment_visible_method_instances.go](../backend/internal/service/payment_visible_method_instances.go)
  - 这意味着“新增中国钱包 provider 并复用现有入口”比较顺。
  - 但“新增新的 first-class 国际卡可见支付方式”会更贵。
- Stripe 还有较明显的专用前端流程，见 [../frontend/src/components/payment/StripePaymentInline.vue](../frontend/src/components/payment/StripePaymentInline.vue)
  - `publishableKey`
  - `clientSecret`
  - Payment Element
  - popup / route 分支

这一层的结论是：`系统允许新增 provider，但不适合一次性推翻现有支付语义`。

## 替换复杂度评估

### A. 新增中国钱包通道并复用现有 `alipay / wxpay` 入口：中等

这是最推荐的路径。

为什么是中等：

- 好处：
  - 后端已有 Provider 抽象、工厂、实例配置、订单快照机制。
  - 后端已有 `alipay / wxpay` 可见方式选择逻辑，可以继续沿用现有用户端入口。
  - 前端无需新增新的支付方式名称，主要改后台配置和路由选择。
- 成本：
  - 需要新增 provider 实现。
  - 需要扩展 `factory.go`、provider 配置校验、管理页字段定义。
  - 如果新渠道要求独立 webhook path，需要补路由和 webhook handler。
  - 需要补 provider 的 `创建支付 / 查单 / 回调验签 / 退款` 测试。

适用结论：

- 这是当前项目最现实的第一步。

### B. 替换 Stripe 的国际卡/订阅购付费链路：中高

这是第二阶段才该考虑的事。

为什么比 A 更贵：

- 当前项目里 Stripe 不只是“一个 provider”，还是一套专用前端交互。
- 现有配置里已经单独暴露 `stripe_publishable_key`，见 [../frontend/src/types/payment.ts](../frontend/src/types/payment.ts) 和 [../backend/internal/handler/payment_handler.go](../backend/internal/handler/payment_handler.go)。
- 前端已经围绕 Stripe 的 `client_secret + Payment Element + popup/redirect` 建好了一条专用链路，见 [../frontend/src/components/payment/StripePaymentInline.vue](../frontend/src/components/payment/StripePaymentInline.vue)。
- 如果替换成新的国际卡渠道，通常需要决定：
  - 继续走前端嵌入式支付表单
  - 改成 Hosted Checkout
  - 还是统一变成后端返回 `pay_url`
- 这不是单纯“换密钥”，而是要重新抽象支付结果结构和前端支付启动逻辑。

适用结论：

- 只有在新渠道明确支持国际卡、退款、查单、Webhook，并且你决定长期淡化 Stripe 依赖时，才值得做。

### C. 全面替换现有 `easypay / alipay / wxpay / stripe`：高

这是不推荐的一步到位路线。

为什么高：

- 会同时影响：
  - provider factory
  - admin provider 配置
  - webhook 路由
  - 前端 visible method
  - 支付流程判断
  - 测试用例
  - 已有订单兼容与退款
- 当前系统并不是完全配置驱动，仍保留不少基于 provider key 和 payment type 的显式分支。
- 如果一轮里同时改中国钱包、国际卡、订阅、退款和后台配置，排查问题会很慢。

适用结论：

- 除非你准备做支付子系统专项重构，否则不建议这么做。

## 推荐迁移顺序

### Phase 1：新增 provider，优先承接中国钱包

目标：

- 新增 `PingPong` 或 `Oceanpayment` provider。
- 中国钱包走新 provider。
- 用户侧继续看到现有 `alipay` / `wxpay`。
- 国际卡先保留 Stripe。

这样做的好处：

- 改动集中在后端 provider 层和后台配置层。
- 前端用户支付入口基本不变。
- 可以先验证新渠道的审核、结算、通知稳定性。

### Phase 2：确认国际卡能力后，再决定是否替换 Stripe

只有在新渠道确认下面几项能力后，才建议评估替换 Stripe：

- 国际卡 Hosted Checkout 或等价方案
- recurring / card-on-file 能力
- Webhook 稳定性
- Refund / dispute 处理能力
- 对账与结算体验

如果这些能力不能一次性确认清楚，就继续保留 Stripe 作为国际卡链路。

### 不推荐路径：首版追求单通道统一

如果你的真实目标是“尽快上线”，那最应避免的是：

- 首版同时换中国钱包和国际卡
- 首版同时重写前端支付交互
- 首版同时替换后台 provider 配置模型

这会把上线节奏从“新增并切流”变成“支付子系统重构”。

## 后续实施检查项

真正开始接入前，建议先让候选渠道给出以下书面确认：

1. 是否支持你的业务类目：`API 服务 / 额度充值 / 月套餐 / 虚拟权益`
2. 是否支持中国钱包与国际卡同时收款
3. 是否支持退款、部分退款、异步通知重试
4. 是否支持未来的 recurring 或 card-on-file
5. 是否需要独立回调地址、白名单、证书或客户端签名
6. 结算周期、保证金、拒付、风控冻结规则

如果第 1 条和第 4 条没有明确答复，不建议直接进入替换 Stripe 的阶段。

## 最终建议

对当前项目，最优路径不是“选一个全新渠道彻底替换一切”，而是：

- 先新增 `PingPong`
- 复用现有 `alipay / wxpay` 用户入口
- 先不动 Stripe 国际卡链路
- 等中国钱包收款稳定后，再评估是否把部分国际卡能力从 Stripe 迁出

这条路的核心价值是：`先解决业务覆盖，再降低系统风险`。
